## Contributing to HAKOMONO

このドキュメントは社内向けの開発ルールと貢献フローを簡潔にまとめたものです。

### 目的
- 複数人で作業する際の基本ルールを明確にします。

### ブランチ戦略
- `main`: 常に安定。デプロイ可能な状態を保つ。
- 機能追加は `feat/<short-description>`、バグ修正は `fix/<short-description>` の形式でブランチを切る。

### コミットメッセージ
- 簡潔に変更点を記述。例: `feat: add README and setup docs`、`fix: handle missing column in source data`。

### プルリクエスト(Review)
- PR を作成したらレビュー担当者に割り当て、少なくとも1人の承認を得てからマージする。
- 小さな変更でも PR を使って履歴を明確に残す。

### テスト / 検証
- 変更前後で手元で動作確認（Streamlit の起動、主要スクリプトの実行）を行う。
- 依存を更新した場合は `requirements.txt` の更新と動作確認を忘れない。

### Issue の立て方
- 問題の再現手順、期待される挙動、実際の挙動、関連ログを添えて Issue を作成する。

### コードスタイル・Lint
- 簡易: 主要な Python ファイルは PEP8 を意識する。必要なら CI で自動チェックを導入する。
