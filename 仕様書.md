# Hakomono プロジェクト全体仕様書 および 依存関係詳細

最終更新日: 2026-04-17

## 1. プロジェクト全体仕様

### 1.1 概要
**Hakomono** プロジェクトは、社内の受注データ（Excelの受注集計表）を読み取り、特定の条件に基づいてデータを抽出・分類し、所定のフォーマット（Excelマクロ有効テンプレート `集計表YYMMDD.xlsm`）へ自動的に転記・保存する Streamlit 製の Web アプリケーションです。
各種の集計作業を自動化し、手作業によるミスを削減することを目的としています。

### 1.2 主要機能
1. **ファイルのアップロードと読み込み**
   - Streamlit の UI を通じて、元データファイル（`受注集計表（上書き用）.xlsx` など）と、書き込み先のテンプレート（`.xlsm`）をアップロードします。
2. **データの抽出・加工機能**
   - 商品データの中で、種類（「箱/こ/不」列）が「箱」となっているものを抽出。
   - 得意先コードが「900以上」のデータは集計対象外として除外（除外ルール）。
   - 「ﾗﾐ(こだ)」と「それ以外（ラミヤマトその他）」の2つのグループにデータを振り分け。
3. **並び替え（ソート）機能**
   - **商品**: 商品コードの昇順。
   - **ラミ（こだ）の得意先**: 得意先コードの昇順。
   - **ラミヤマトの得意先**: 優先順位リスト（`"ﾗﾐ"`, `"ﾗﾐ(ｻﾈｯﾄ行)"`, `"桜花便"`, `"社内便1"`, `"社内便2"`, `"社内便3"`）に従ってソート。
4. **テンプレートへの動的書き込み（Excel 操作）**
   - 既存の Excel テンプレート（マクロを含む）を壊さずに、各シート（「ラミ（こだ）」、「ラミヤマトその他」）の特定の行（ヘッダー・フッター・データ行）に必要な行数を挿入し、正確なセル位置へデータを書き込みます。
5. **ファイルのダウンロード**
   - 処理が完了したファイルを新しい名前（`集計表YYMMDD.xlsm`）でダウンロード提供します。

## 2. `.py` ファイル単位の依存関係と詳細解説

プロジェクト内に存在する主要な `.py` ファイルについて、各ファイルがどのように連携し、何に依存しているかを精査・解説します。

### 2.1 `app.py`
ユーザーインターフェース（UI）およびアプリケーションのメインエントリーポイントとなるファイルです。

- **役割**: Streamlit を用いたWebインターフェースの構築。ユーザーからのファイル受け付け、`logic.py` 内への処理の委譲、及び処理済みファイルのダウンロード提供を担います。
- **外部ライブラリ依存**:
  - `streamlit`: UI要素（タイトル、アップローダー、ボタン、スピナー、ダウンロード機能等）の構築。
  - `pandas`: 処理結果の一部を UI（展開形のプレビュー機能）で表示するため。
- **内部モジュール依存**:
  - **`logic.py` に強く依存**:
    - `load_source_data` (ソース読み込み)
    - `process_data` (データの抽出・振り分け・ソート)
    - `write_to_template` (Excelテンプレートへの書き込み用ByteIO生成)
- **ポイント**: 独自のビジネスロジックは一切含まず、UIの処理から `logic.py` へデータを橋渡しする「コントローラー」の役割を果たしています。

### 2.2 `logic.py`
プロジェクトの心臓部であり、データ加工ロジックおよびExcelのI/O処理がすべて集約されているファイルです。

- **役割**: データの読み込みルール、不要な列の除外ルール、特殊なソート処理、およびテンプレートへの直接書き込み機構（Excelの操作）の実装。
- **外部ライブラリ依存**:
  - `pandas`: Excelファイルの読み込み、DataFrameによる行・列操作とデータの統合・分離。
  - `openpyxl`: Excelテンプレート(`.xlsm`)への書き込み。既存のマクロ（`keep_vba=True`）を維持したまま、特定のシートからヘッダーとフッターを検索し、動的に行を挿入・クリア・上書きする高度な処理を行います。
- **標準ライブラリ依存**:
  - `re`: 正規表現。得意先コードの数字部分を抽出してソートキーを作成（`_get_sort_key`関数）するために利用。
  - `io`, `datetime`: メモリ上でのファイルストリーム（`io.BytesIO`）展開やファイル名の日付生成。
- **内部モジュール依存**:
  - なし（**他からの呼び出し元として機能する完全な独立モジュール**）。
- **ポイント**: 他のどのソースファイルにも依存しないため、UI（Streamlit）から完全に切り離して内部の単体テストが可能な構造になっています。

### 2.3 `verify_koda.py`
ロジック検証（テスト）用のスクリプトです。「ラミ（こだ）」向けの処理が仕様通りに動くかを検証します。

- **役割**: `logic.py` の関数群をローカルファイル（`受注集計表（上書き用）.xlsx`, `集計表251117.xlsm`）に擬似アプリケーションとして適用し、結果を検証（アサート）します。
  「対象外コード（900以上）の除外」「商品コード／得意先コードのソート」「2行ヘッダー構成による正確なExcel出力」が正しく行われたかをテストし、標準出力します。
- **外部ライブラリ依存**:
  - `pandas`, `openpyxl`: テスト結果であるデータフレームの検証や、出力されたExcelファイルの中身・セルを再読み込みして検証するため。
- **内部モジュール依存**:
  - `logic.py`: テスト対象として `load_source_data`, `process_data`, `write_to_template` をインポートします。
- **ポイント**: Streamlitを起動せずとも、バックエンドのロジックが破綻していないかを即座に確認（コンソール出力 `PASS`/`FAIL`）できます。

### 2.4 `verify_yamato.py`
ロジック検証（テスト）用のスクリプトです。「ラミヤマトその他」向けの処理が仕様通りに動くかを検証します。

- **役割**: `verify_koda.py` と同様に `logic.py` の動作確認を行いますが、こちらは「独自の優先順位による得意先ソート（ﾗﾐ → ﾗﾐ(ｻﾈｯﾄ行) → 桜花便...）」、「3行ヘッダー構成（配送種別、得意先名）の正確な出力」に重点を置いてテストを実施します。
- **外部ライブラリ依存**:
  - `pandas`, `openpyxl`: 同上（テスト結果の自動検証等）。
- **内部モジュール依存**:
  - `logic.py`: テスト対象としてメインロジック関数に依存。
- **ポイント**: `logic.py` 内部でもヤマト用の出力は行挿入やヘッダーの扱いが「こだ用」と大きく異なるため、個別にテストファイルが分離・用意されています。

## 3. ファイル間の依存関係図（ディペンデンシー・アーキテクチャ）

```mermaid
graph TD;
    subgraph UI モジュール
        A[app.py]
    end

    subgraph バックエンド・ロジック
        B[logic.py]
    end

    subgraph 検証・テストスクリプト
        C[verify_koda.py]
        D[verify_yamato.py]
    end
    
    subgraph 外部ライブラリ
        L1(streamlit)
        L2(pandas)
        L3(openpyxl)
    end

    %% UIからの依存関係
    A -- import --> B
    A -- import --> L1
    A -- import --> L2

    %% テストの依存関係
    C -- import --> B
    D -- import --> B
    C -. assertで利用 .-> L2
    C -. assertで利用 .-> L3
    D -. assertで利用 .-> L2
    D -. assertで利用 .-> L3

    %% ロジックの依存関係
    B -- import --> L2
    B -- import --> L3
```

## 4. 総括
本プロジェクトは、`logic.py` にデータ処理およびファイル入出力の全てのドメインロジックが凝集されている、責務分解の優れたアーキテクチャになっています。
UIを提供する `app.py` や、処理の正当性を自動検証する `verify_*.py` のすべてが共通の `logic.py` を呼び出す形になっているため、今後仕様変更（抽出条件やソートの優先順位変更など）があった場合は `logic.py` さえ修正すればプロジェクト全体の整合性が保たれる仕組みです。
